Was ist ein Datenprodukt in Signal?

Datenprodukt ≠ Dataset ≠ Contract

Ein Datenprodukt ist das Ganze — alle Objekte einer einzigen Ownership über alle Layer. Ein Contract beschreibt nur die Ränder. Ein Dataset ist ein einzelnes technisches Objekt.

BegriffWas es ist
DatenproduktEigentums-Aggregat über alle Layer
ContractQualitätszusage an einem Output-Port
DatasetEinzelnes Datasphere/HANA-Objekt
GuaranteeSemantische Qualitätseigenschaft (kein SQL)
CheckCompiliertes, ausführbares SQL-Template

Kernprinzip

„Ein Data Contract ist nur der Rand des Datenprodukts — das Versprechen an der Konsumgrenze. Das Innere ist interne Angelegenheit." — ADR-0001

Signal übersetzt SQL-freie, semantische Contracts deterministisch in read-only SQL-Checks, die gegen HANA/Datasphere laufen. Ergebnisse fließen ins Cockpit als Compliance-Ampel, Lineage-Map und Incidents.

Das Interieur (alle Objekte bis auf den Port) wird nicht deklariert — es wird aus der Lineage automatisch abgeleitet. Layer ist eine Beschreibung, keine Grenze.

Die fünf Bausteine

📦

Manifest

products/<name>.yaml — deklariert nur Identität, Owner-Hülle und Output-Ports. Kein Interieur.

📋

Contract

contracts/<dataset>.yaml — semantische Guarantees am Output-Port. Kein SQL.

⚙️

Compiled Checks

checks/<dataset>/checks.yaml — deterministisch aus dem Contract compilierte SQL-Templates.

🔗

Lineage-Graph

data/lineage.json — Quelle für das abgeleitete Interieur und die Reconciliation-Befunde.

📊

Result-Store

SQLite/HANA — persistiert Check-Ergebnisse, Compliance-Status, Incidents, Baselines.

🚦

Cockpit

React-Frontend — Compliance-Ampel, Coverage-Map, Contract Workbench, Incidents.

Determinism Hash

    
Jede Compilation wird mit einem reproduzierbaren SHA256-Hash gestempelt:

    
compiler_hash = sha256(f"{contract_hash}:{library_version}")[:16]

    
Derselbe Contract + dieselbe Library-Version → immer derselbe Hash. Der Hash wird in die erzeugte checks.yaml eingebettet und ermöglicht auditierbare, deterministische Nachvollziehbarkeit.

  







  
Produkt-Komposition: Manifest + abgeleitetes Interieur


  

    

      
Das Manifest — dünne Deklaration

      
Nur Identität, Owner-Hülle und Ports. Das Interieur fehlt bewusst.

      
product: sales_overview
owners: [team-sales]

output_ports:
  - dataset: DS_REVENUE_SUMMARY

inbound:
  - depends_on:
      product: kunde
      version: "1.2.0"

# interieur: NICHT gelistet
# → aus Lineage abgeleitet

    

    

      
Owner-gegateter Upstream-Walk

      
Signal läuft vom Output-Port rückwärts durch den Lineage-Graphen:

      
    
        
  1. Start am Output-Port (DS_REVENUE_SUMMARY)
    2. 
        
  2. Rückwärts durch alle Upstream-Kanten
    3. 
        
  3. Stopp bei (a) Port eines anderen Owner-Sets
    4. 
        
  4. Stopp bei (b) externem Source-Knoten
    5. 
        
  5. Stopp bei (c) bereits besuchtem Knoten
    6. 
        
  6. Alles dazwischen = Interieur
    7. 
      

      
Layer ist Beschreibung, keine Grenze. Owner-Set entscheidet die Zugehörigkeit.

Compliance der eigenen Outbound-Contracts. Das ist die Produkt-Ampel — governance-relevant.

Compliance der depends_on-Produkte. Bricht kunde, wird sales_overview nicht automatisch rot — es erhält ein Risiko-Signal. Separate Spur, nicht vermischt.

      

    

  







  
Konvergenz mit Data Product Studio (BDC)

  

    Hinweis: Die genaue technische Struktur von Studio-Produkten ist teilweise unverifiziert (V2/V5 aus ADR-0003 offen). Annahmen sind als solche gekennzeichnet.
  


  

    

      
Was sich NICHT ändert — Konzept-Ebene

      
Speicher-agnostisch. Gilt 1:1 für Studio-Produkte:

      
    
        
  • boundary-Klassifikation (internal/inbound/outbound)
    • 
        
  • Tiering-Modell (Tier 0/1/2 × Lite/Full)
    • 
        
  • Manifest-Modell (dünne Deklaration)
    • 
        
  • Compliance-Ampel und Incident-System
    • 
        
  • Gekettete Contracts (depends_on)
    • 
        
  • ORD/ODCS als einseitige Derivate
    • 
      

    

    

      
Was sich ändert — Enforcement-Ebene

      
Signal enforced nur an SQL-Oberflächen:

      
GX-on-HANA (hdbcli, read-only)
↓
SQL-Template gegen
  "{schema}"."{dataset}"
↓
20 Bibliotheks-Checks

      
Kein zweiter Executor. Ein Spark-/Delta-Executor wird abgelehnt (G7, single executor, ENGINE-FROZEN).
