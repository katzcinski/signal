# PLAN — Managed Service (Instanz-pro-Tenant) umsetzbar machen (v1)

**Stand:** 2026-06-25 · **Status:** Vorschlag / Planung (noch kein Code)
**Kontext:** Konkretisiert die Lücken aus
[`Konzept_Managed_Service_Provisioning.md`](Konzept_Managed_Service_Provisioning.md)
§5 — **zugeschnitten auf das tatsächlich gewünschte Betriebsmodell A1
(Instanz-pro-Tenant)**, nicht auf gepoolten Multi-Tenant-SaaS.

## Scope-Entscheidung (verbindlich für diesen Plan)

| | Im Scope | Begründung |
|---|---|---|
| **Produktiver `HanaStore`** | ✅ | Der eigentliche Blocker. Ohne ihn läuft Managed nur auf SQLite-pro-Tenant. |
| **Provisioning-Automation** | ✅ | „Tenant anlegen" muss wiederholbar/fehlerarm sein, sonst skaliert Betrieb nicht. |
| **Pro-Tenant Konfig-/Secret-Härtung** | ✅ | Saubere Isolation ohne App-Umbau. |
| **`tenant_id` / Row-Level-Pooling** | ❌ **bewusst de-scoped** | Schiebt Signal von Dienstleistung → Softwareprodukt (`Uebergabemodelle_und_Lizenz.md` §1). Instanz-pro-Tenant isoliert stärker bei minimalem Eingriff. |

> Leitsatz: **Isolation über Infrastruktur, nicht über Query-Filter.** Solange
> das gilt, bleibt der Code einfach und die Delivery-Posture sauber.

## Arbeitspaket A — Produktiver `HanaStore`

### Befund
`packages/dq_core/store/hana_store.py` ist ein **Stub** (alle Methoden
`raise NotImplementedError`). Er deckt nur die **17 Methoden** des
`ResultStoreProtocol` (`base.py`) ab. `SqliteStore` ist aber auf **~49
öffentliche Methoden** gewachsen — Incidents (`open_incident`,
`transition_incident`, `auto_resolve_incidents`, `list_incidents`), Schedules
(`claim_due_schedules`, `create_schedule` …), Notifications (Channels/Rules/
Mutes), SLA, Compliance-Events, Object-Status-Rollups. **Das Protocol ist
veraltet** und unterzeichnet die reale Store-Fläche.

### Entscheidungen (festgelegt 2026-06-26)
- **Methoden-Fläche: voller Gleichstand.** `HanaStore` deckt für Managed-v1 die
  **gesamte ~48-Methoden-Fläche** von `SqliteStore` ab (Results/Runs, Operations,
  Compliance, Diagnostics, `try_begin_run`, **Incidents, Schedules, Notifications,
  SLA, Object-Status-Rollups**). Begründung: kein degradiertes Cockpit pro Tenant
  — Managed soll funktional deckungsgleich zur SQLite-Variante sein.
- **Migrationen: dialekt-getrennte Dateien** (Option (a)). Kein
  Laufzeit-Übersetzungs-Layer.
- **Doppellauf-Guard: HANA Filtered Index** als Äquivalent zum SQLite-Partial-Index.
- **CI-Verifikation: echte HANA im (nicht-blockierenden) Optional-Job** — adressiert
  das benannte Hauptrisiko (SQL-Dialekt-Drift), das ein reiner Mock nicht abfängt.

### Vorschlag
1. **Protocol aktualisieren** (`base.py`) auf die **volle real konsumierte Fläche**
   (alle ~48 öffentlichen Methoden, nicht die historischen 17). Quelle der Wahrheit
   ist die `SqliteStore`-Oberfläche, die die API-Router tatsächlich aufrufen — inkl.
   `try_begin_run`, `get_compliance_events`, `get_all_runs`, `get_latest_run`,
   `get_object_status`, `get_object_family_status`, `get_sla` sowie der Incident-/
   Schedule-/Notification-Familien. Konformitätstest entsprechend erweitern, sodass
   beide Stores denselben (vollständigen) Vertrag erfüllen.
2. **`HanaStore` implementieren** gegen `dq_results_lt`, methodengleich zu
   `SqliteStore` (voller Gleichstand, s. o.). SQL-Dialekt-Unterschiede sauber
   kapseln:
   - `INTEGER PRIMARY KEY AUTOINCREMENT` → HANA `GENERATED BY DEFAULT AS IDENTITY`.
   - `INSERT OR REPLACE` / `INSERT OR IGNORE` → HANA `UPSERT … WITH PRIMARY KEY` /
     `MERGE`.
   - Partial Unique Index `idx_dq_runs_one_running` (Doppellauf-Guard) → **HANA
     Filtered Index** (entschieden), DB-seitig wie heute, trigger-frei.
   - Zeitstempel/Boolean-Mapping (`_ts`, `_num`-Helfer) angleichen.
3. **Migrationen HANA-fähig machen — dialekt-getrennte Dateien** (entschieden):
   parallele `migrations/NNN_*.hana.sql` neben den bestehenden SQLite-Dateien.
   Explizit, testbar, kein verstecktes Rewrite; geshippte SQLite-Migrationen bleiben
   unangetastet. Der Migrations-Runner wählt den Dialekt-Satz anhand des aktiven
   Backends.
4. **Konformitätstest** in `tests/unit/`: `HanaStore` erfüllt dasselbe (volle)
   Protocol wie `SqliteStore` — die Form-Konformität läuft gegen einen
   `hdbcli`-Mock und ist **blockierend** in jedem PR.
5. **Echter HANA-Integrationstest als Optional-Job** (entschieden): nicht-blockierender
   CI-Job gegen eine HANA-Test-Instanz, der die Migrationen fährt und die
   Dialekt-kritischen Pfade (UPSERT/MERGE, IDENTITY, Filtered-Index-Guard,
   Doppellauf bei ≥2 Workern, F2) real ausführt. Form-Konformität (Mock) ≠
   Verhaltens-Korrektheit (echte HANA) — beide werden gebraucht.

### Betroffene Stellen
- `packages/dq_core/store/hana_store.py`, `base.py` (Engine, **G7** framework-frei).
- `packages/dq_core/store/migrations/` (dialekt-getrennt, **neue** Dateien).
- `services/api/deps.py` — `STORE_BACKEND=hana` darf nicht mehr hart abweisen,
  sobald `HanaStore` real ist (heute bewusster `raise`).
- `tests/unit/` Store-Konformität.

### Aufwand / Risiko
**Groß** — durch den entschiedenen **vollen Methoden-Gleichstand (~48 Methoden)**
bewusst am oberen Ende; größtes Paket, aber der eigentliche Freischalter für
Managed und ohne Cockpit-Feature-Lücke pro Tenant. Restrisiko nach den
Festlegungen:
- **SQL-Dialekt-Drift** — wird durch den echten HANA-Optional-Job abgefangen
  (nicht durch den Mock); der Mock sichert nur die Protocol-Form.
- **Doppellauf-Guard (F2)** — Korrektheit bei ≥2 Workern muss auf HANA via
  Filtered Index genauso halten wie auf SQLite; explizit im HANA-Integrationsjob
  zu testen.
- **Pflege-Duplikat** der dialekt-getrennten Migrationen — jede neue Migration
  braucht künftig SQLite- *und* HANA-Datei; im Migrations-PR-Review prüfen.

## Arbeitspaket B — Provisioning-Automation (Instanz-pro-Tenant)

### Befund
„Tenant anlegen" ist heute Handarbeit: Schema/DB, OIDC-Client, Git-Repo,
`environments.yml`, Secrets, Deployment. Fehleranfällig und nicht auditierbar.

### Vorschlag
Ein **deklaratives Tenant-Manifest** + ein idempotentes Provisioning-Skript
(`scripts/provision_tenant.py`), das aus einem Manifest den vollständigen Stack
herstellt:

```
tenant:
  id: kundeA
  oidc: { issuer: …, audience: kundeA, role_mapping: { … } }
  store: { backend: hana, schema: DQ_RESULTS_KUNDEA }   # oder sqlite-Volume
  git_remote: git@…/kundeA-contracts.git
  environments_ref: vault:kundeA/environments            # Secret-Referenzen
  executor: hosted | hybrid                              # §3c
```

Schritte (idempotent, jeder einzeln wiederholbar):
1. Store bereitstellen (HANA-Schema anlegen + Migrationen fahren / SQLite-Volume).
2. Git-Repo anlegen/verknüpfen, leere `contracts/`-Struktur seeden.
3. OIDC-Client + Rollen-Mapping registrieren.
4. `environments.yml` aus Vault-Referenzen rendern (nie Klartext-Secrets).
5. Deployment-Unit ausrollen (Namespace/Pod, `settings`-ENV aus dem Manifest).
6. Smoke-Check: `/health`, Bind-Policy (S5), `allow_mock_connection=false`.

### Betroffene Stellen
- `scripts/provision_tenant.py` (neu), Manifest-Schema (neu).
- Deployment-Templates (K8s/Compose) parametrisiert je Tenant.
- Kein Eingriff in `dq_core`.

### Aufwand / Risiko
**Mittel.** Reine Orchestrierung um bestehende Konfig-Achsen. Risiko gering, wenn
strikt idempotent und secret-frei (nur Referenzen).

## Arbeitspaket C — Pro-Tenant Konfig-/Secret-Härtung

### Befund
`secrets.py` ist „vault-ready" (Protocol `SecretResolver`), nutzt aber heute den
Env-Resolver. Für viele Tenants sauber: pro Tenant gescopte Secrets statt eines
globalen Env-Raums.

### Vorschlag
- Einen `VaultSecretResolver` (HashiCorp Vault **oder** BTP Credential Store)
  hinter dem bestehenden Protocol implementieren — Consumer bleiben unverändert,
  nur der Default-Resolver wird pro Deployment getauscht.
- Secret-Pfade pro Tenant namespacen (`<tenant>/hana_pw`, `<tenant>/oidc_secret`).
- Bestätigen, dass nach außen weiterhin nur ein boolescher `status` geht, nie der
  Wert (ist im Resolver bereits Disziplin).

### Betroffene Stellen
- `services/api/secrets.py` (neuer Resolver, Protocol unverändert).
- `tests/api/` (Resolver-Auswahl, kein Secret im Response/Log).

### Aufwand / Risiko
**Klein–mittel.** Additiv, durch das bestehende Protocol abgesichert.

## Reihenfolge & Abhängigkeiten

```
A (HanaStore) ──┬─▶ schaltet Managed auf HANA frei (sonst nur SQLite-pro-Tenant)
                │
B (Provisioning)─┴─▶ braucht A für den Store-Schritt, C für den Secret-Schritt
C (Vault-Secrets) ──▶ unabhängig startbar, B nutzt es
```

Empfehlung: **C parallel** (klein, sofort nützlich) · **A zuerst inhaltlich**
(Freischalter) · **B zuletzt** (bündelt A+C zu „ein Kommando = ein Tenant").

## Gate-Checkliste

| Gate | Beachten |
|---|---|
| **G7** `dq_core` framework-frei | A: HanaStore/Protocol importieren kein FastAPI. |
| **Migrationen** | A: HANA-Migrationen als **neue** Dateien; SQLite-Migrationen nie ändern. |
| **S5** Fail-closed Bind | B: Smoke-Check erzwingt non-loopback nur mit Auth. |
| **S-13** kein Fail-Open | B: `allow_mock_connection=false` im Manifest erzwingen. |
| **G8** PII-Gate | A: Diagnostics-Insertion bleibt hinter `_allow_diagnostics`. |
| Secret-Disziplin | C: nur Referenzen, Wert nie geloggt/serialisiert. |

## Bewusst NICHT in diesem Plan

- **`tenant_id` / gepooltes Multi-Tenant** — Scope-Entscheidung oben: würde die
  Dienstleistungs-Posture aufgeben. Falls je gewünscht, ist es ein **eigenes
  Projekt** mit Store-Achse, Token-basierter Mandantenauflösung und
  durchgängigem Query-Scoping — nicht Teil von Managed-Service-A1.
