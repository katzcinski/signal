Wie Signal denkt & arbeitet

Signal verwandelt semantische Garantien über Datasphere-Objekte in deterministisch kompilierte, ausführbare Quality-Checks, fährt sie ausschließlich lesend gegen HANA und macht den Zustand als Cockpit, Compliance-Ampel und Coverage-Map sichtbar. Diese Seite zeigt die Mechanik — interaktiv, vom Contract bis zum Lauf.

Kein SQL im Contract (G1) HANA nur lesend Deterministischer Compiler Ein Code · zwei Deployments

Tipp: Die Kästen weiter unten sind anklickbar. Das Compiler-Labor kompiliert echte Garantien live im Browser — mit derselben Logik wie dq_core/contract/compiler.py.

01 · Die Denkweise

Vier Leitprinzipien, die alles erklären

Bevor man die Mechanik versteht, muss man die Haltung dahinter verstehen. Jede Designentscheidung in Signal lässt sich auf einen dieser vier Sätze zurückführen.

A Contracts tragen Garantien, nie SQL

Ein Contract ist SQL-freies YAML — er beschreibt was gelten soll (Schlüssel eindeutig, Daten frisch, Spalte vollständig), nicht wie man es misst. Der einzige Ort, an dem SQL entsteht, ist der Compiler. So darf ein Fachverantwortlicher Zusagen formulieren, ohne je SQL zu schreiben — und SQL-Injection über Contracts ist strukturell ausgeschlossen (Gate G1).

B Die Engine ist eingefroren & frameworkfrei

dq_core importiert nie FastAPI. Die Check-Engine und die Expectation-Grammatik sind [ENGINE-FROZEN]: man erweitert über die Check-Bibliothek, statt die Engine zu ändern. Dadurch läuft derselbe Kern in der API, in der CLI und im Cron-Job — und Regressionen werden von der bestehenden Test-Suite verboten (Gate G5/G7).

C Drei Zustands-Achsen, strikt getrennt

Ein Objekt hat nicht einen Status, sondern drei unabhängige: Lifecycle (Erstellung) · Compliance (hält die Zusage?) · Coverage (wie gut abgedeckt?). Sie zu vermischen ist der häufigste Denkfehler in DQ-Tools — Signal hält sie konsequent auseinander.

D Ein Code, zwei Deployments

Berater-lokal (SQLite, NoAuth, 127.0.0.1) und Kunde (HANA, OIDC, Multi-Worker) laufen aus demselben Code — über eine Auth- und Store-Abstraktion, ohne Code-Zweige. Was lokal grün ist, ist beim Kunden identisch verdrahtet.

Merksatz: Garantie ist semantisch · Check ist generiert · Schema wird erst zur Laufzeit gebunden · Compliance lebt nur im Store. Diese vier Trennungen ziehen sich durch jede Zeile.

06 · Drei Achsen, drei Maschinen

  
Lifecycle · Compliance · Coverage

  
Der Kern des Denkmodells, ausbuchstabiert. Drei unabhängige Zustandsräume — keiner darf den anderen überschreiben.

  

    

      
 Lifecycle · im YAML / Git

      
Erstellungszustand. PUT erzeugt immer einen Draft; approve/certify aktivieren.

      

      
draft → active → deprecated

      
Niemals compliance! Lifecycle sagt nichts darüber aus, ob die Zusage gehalten wird.

    

    

      
 Compliance · nur im Store

      
Hält die aktive Zusage? Abgeleitet aus dem letzten Lauf, lebt ausschließlich im Result-Store (A1).

      

      
unknown → compliant ⇄ breached

      
breached bei ≥1 nicht bestandenem Check mit Severity ∈ {fail, critical}. Auto-Recovery beim nächsten grünen Lauf. Jeder Übergang ist ein Event; ein Neu-Breach öffnet ein Incident.

    

    

      
 Coverage · abgeleitet

      
Wie gut ist ein Objekt überhaupt abgedeckt?

      

      

        covered active + Checks

        partial active, keine Checks

        gap Contract, nicht active

        out_of_scope kein Contract

07 · Zusage halten

  
Compliance, SLA & Incidents

  
Wie aus Check-Ergebnissen ein verbindlicher Zustand wird — und wie eine Verletzung zu einer nachverfolgbaren Episode wird.

  

    

      

        
Die Regel (v1)

        
# dq_core/contract/compliance.py
BREACHING_SEVERITIES = {"fail", "critical"}

def compute_compliance(results):
  if not results: return "unknown"
  for r in results:
    if r.severity in BREACHING_SEVERITIES \
       and not r.passed:
      return "breached"
  return "compliant"

        
warn zählt nie zum Breach — Warnungen sind informativ, nicht vertragsverletzend.

      

    

    

      

        
Vom Ergebnis zum Incident

        

          
Lauf grün → compliance bleibt/wird compliant, since nur bei Wechsel aktualisiert

          
Lauf rot → breached: Event geloggt (from→to), Incident geöffnet

          
Incident → open → acknowledged → investigating → resolved, mit Timeline je Aktion

          
Recovery → grüner Folgelauf ⇒ Auto-Recovery, Incident wird aufgelöst

        

        
SLA-Fenster (GET …/sla) rechnen über die Übergangs-Zeitstempel: weil since die letzte Transition markiert, lässt sich der Compliant-Anteil über 7/30/90 Tage exakt bestimmen.

      

    

  











  
08 · Der Kreis schließt sich

  
Observability & der Proposal-Miner

  
Signal beobachtet nicht nur — es lernt aus der Historie und schlägt schärfere Garantien vor. Operative Daten fließen zurück in Governance (Shift-Left).

  

    

      
 Rolling-Baselines

      
Pro (dataset, metric) werden laufende Statistiken geführt: mean, stddev, p01/p99, robuster MAD. In der Warm-up-Phase bestehen DELTA-Expectations automatisch — kein Fehlalarm ohne Basis.

    

    

      
 Miner

      
Aus der Actual-Value-Historie eines Checks leitet der Miner Vorschläge ab, typisch BETWEEN p01 AND p99 — mit Evidenz (n, min/max, mean, stddev) und einem Confidence-Wert, der mit der Stichprobengröße wächst.

    

    

      
 Steward entscheidet

      
Vorschläge landen als Inbox: accept / reject / snooze. Ein Accept patcht die Garantie im Contract-YAML → nächster Compile nutzt die neue Schranke. Der Regelkreis schließt sich.

    

  

  

    Loop: Lauf-Historie → Miner → Proposal (BETWEEN p01 AND p99) → Steward accept → Contract-Patch → Compile → schärferer Check

11 · Die Oberfläche

  
API — was man anfassen kann

  
FastAPI unter /api, RFC-7807-Fehler, OpenAPI unter /api/openapi.json (das Frontend zieht daraus seine Typen, Gate G4). Eine Auswahl der zentralen Ressourcen:

  

  

    

      
Lite vs. Full

      

        
LiteFull

        
Aktivierung…/certify (1 Schritt)PUT→approve→compile

        
Versionierungkeine PflichtSemVer, Breaking⇒Major

        
Breaking-Gatenur f. zertifizierteimmer blockierend

      

      
Beide Modi teilen Engine, Compiler, Bibliothek und die scharfen Gates — sie unterscheiden sich nur in der Zeremonie.

    

    

      
Ein Lauf, drei Wege

      
Die API triggert Läufe nur ad hoc (POST …/run → 202 {run_id}). Regelmäßige Läufe plant ein externer Scheduler über die CLI (cli/dq_check_runner.py). Beim Kunden schützt eine Run-Registry im Store gegen Doppelläufe bei ≥2 Workern. Fortschritt streamt live per SSE.
