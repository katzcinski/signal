01 · Overview

The pipeline end-to-end

G1 — No SQL in contracts

Contracts are semantic YAML (guarantees: families). SQL is produced exclusively by the compiler. The schema (CENTRAL etc.) is injected at runtime via bind_schema(…) — never hardcoded. CI greps for literals.

🧱

G6 — States never silently omitted

Every check the engine considers produces a CheckResult row — including those it decided not to execute. Dropping a check is forbidden. Skipped checks are visible with an explicit reason.

02 · Core concept

Check result states

The state of a check result answers "what did the engine decide about this check?" and is orthogonal to the verdict (did the SQL expectation hold?). Only two states count toward status and compliance — the rest are visible but neutral.

03 · Execution flow

Gating — cheap checks gate expensive ones

When gating=True (always on for UI-triggered runs), the engine runs checks in two phases: gates first, then evaluates whether the remainder should run. A failed gate means expensive consistency scans would operate on untrusted data — they're skipped and flagged skipped_stale.

Execution modes

Independently of gating, runs have an execution_mode:

  • batch — all checks in one UNION ALL … FROM DUMMY round-trip
  • isolated — one statement per check; slower but pinpoints failures
  • auto (default) — try batch; if it throws, fall back to isolated

DELTA warm-up

Checks using the DELTA <= N% expectation compare the current actual against the previous run's actual. If no previous value exists (first run, or the previous run errored), the check passes automatically — warm-up is not a failure. Baselines need ≥ 5 samples before statistics are computed.

04 · Status derivation

Status rollup — worst-of

The object status and family status are both worst-of the executed check results from the latest finished run. Only executed and error states contribute — skipped results are excluded before the worst-of calculation.

Severity scoring

1
error
SQL exception, wrong shape, timeout — we don't know the answer
2
warn
Quality issue worth noting; does not breach a contract
3
fail
Business rule broken; breaches contract if under active contract
4
critical
Highest severity — e.g. empty table, schema dropped
0
pass
All executed checks satisfied their expectation

error ranks between warn and fail in the score table, but in _overall_status() it short-circuits first — any exception makes the overall status error regardless of other results.

Two rollup axes

Object status

Single worst-of across all check types. Shown in the status grid, the object list, lineage nodes.

store.get_object_status()
Family status

Same worst-of, but split by family: observability (freshness/volume/schema checks) vs. quality (completeness/consistency). An object can be ● obs and ✕ qual simultaneously.

store.get_object_family_status()
Coverage flag

Third axis for the lineage map: is this object adequately watched? Independent of pass/fail verdict.

05 · Breach flow

Compliance & incidents

After every finished run on an object under an active contract, Signal computes a compliance state and handles transitions. warn failures never breach. Skipped checks don't either.

SLA measurement

The SLA is the percentage of wall-clock time an object spent compliant within a window (7 / 30 / 90 days). It's time-weighted uptime, not a pass-rate per run. Computed from dq_compliance_events transitions.

Contract kinds & breach type

consumer/provider contract → Contract-Breach incident.
internal_gate → Engineering-Signal incident. Both use the same incident table; the kind field distinguishes them.

warn does not breach

Only severities in {"fail", "critical"} count toward a breach. A run where every fail/crit check passes is compliant even if warn checks failed — those are surfaced but don't trigger the incident flow.

06 · Disambiguation

Staleness — three unrelated things

"Stale" appears in three different places in Signal with three different meanings. Don't conflate them.

1

Gate staleness gate

A freshness or volume gate check failed during a run. Expensive checks for that run become skipped_stale.

Property of a single run's results. Nothing persists. The next run starts fresh.

Per-check: CheckResult.state = "skipped_stale"
2

Extract staleness 🗺️

data/lineage.json and the inventory are point-in-time extracts from Datasphere. The lineage endpoint reports extract_age (days since file mtime) and flags stale = age > extract_stale_days (default: 7 days).

Says "the map you're looking at may be out of date." Completely independent of DQ verdicts.

lineage response: { stale: true, extract_age: N }
3

Run staleness 🕐

An object whose latest finished run is old — visible via last_run on the object status. The UI can warn that the green status is based on an old measurement.

No server-side gate. A green object that hasn't been re-checked in weeks stays green — but the timestamp makes the age visible.

ObjectOut.last_run (timestamp)
07 · Downstream impact

Blast radius

Blast radius = everything downstream that a data quality problem can contaminate. Signal computes it via BFS over columnEdges in the lineage graph — enriched with ownership and coverage flags so you know who to tell and where the blind spots are.

Watch a column failure spread through the downstream graph.
Source (failing) Depth 1 (direct) Depth 2 Not yet reached Coverage: ● covered   ◐ partial   ▲ gap   ○ out-of-scope

How the BFS works

Starting from (object, column), the algorithm traverses columnEdges downstream: each reachable (target_object, target_column) is reported once at its minimum hop distance. Cycle-safe via a visited set; bounded by max_depth (default 25). Edges left unexplored at the cap set truncated: true.

Why coverage flags matter in the blast zone

A ▲ gap node in the blast radius is where an upstream defect will silently leak through — no check will catch it reaching consumers. Blast radius enrichment surfaces these blind spots along with the owner so you know exactly who has an unguarded exposure.

The one rule to remember

A check result's state (did it run?) is independent of its verdict (did it pass?). Only executed and error results affect status and compliance. skipped_* and downgraded are visible but status-neutral (G6). That single principle is what makes stale-gating, dependencies, and honest rollups all work without lying to the user.
