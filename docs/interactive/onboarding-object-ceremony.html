Signal · Onboarding, Checks & Ceremony

From object to active guarantee

How an object in Datasphere becomes a measured, versioned, compliance-tracked data product — and what ceremony is required along the way.

Lite

Verbindlichkeit ohne Zeremonie

Measurable guarantees in days, not weeks. One click: save & activate. No SemVer, no approval dialog.

  • Default for internal_gate
  • Owned by platform/consulting team
  • Draft → active in a single /certify call
  • Gates G1, G6, G8 still fully enforced
  • G3 still blocks breaking changes on certified products
Full

Governte Data Products

Versioned, owner-approved promises with a traceable change history.

  • Default for consumer_contract / provider_contract
  • Owned by the product (Fachbereich)
  • draft → diff → approve → compile → run
  • SemVer mandatory; breaking change = major bump
  • G3 always blocking, CI + server-side
  • SLA windows, version history, proposals inbox
Same engine, different ceremony. Both modes share one compiler, one engine, one result store, and the same security gates. The difference is entirely in the process — not in what gets measured.

03 · Contract structure

Contract anatomy

A contract is a semantic YAML file — no SQL ever. The compiler turns it into read-only check SQL at certify/compile time. The schema is never in the contract; it is bound at runtime per environment (G2).

product: FACT_ORDERS # SQL-safe identifier = filename dataset: FACT_ORDERS # HANA table/view kind: internal_gate # or consumer_contract / provider_contract owned_by: platform # or product owners: ["grp:data-platform"] # optional ACL (IdP groups or subs) version: 1.0.0 # SemVer — mandatory in Full mode lifecycle: draft # draft | active | deprecated (never in SQL) guarantees: schema: columns: [ORDER_ID, CUSTOMER_ID, ORDER_DATE, NET_AMOUNT] mode: closed # closed = new columns are a breaking change keys: - columns: [ORDER_ID] unique: true severity: critical freshness: column: ORDER_DATE max_age: PT24H # ISO-8601 duration severity: warn not_null: - columns: [ORDER_ID, CUSTOMER_ID] severity: fail volume: min_rows: 1000 severity: warn

What never goes in a contract (G1)

No sql:, no SELECT, no schema literals, no CENTRAL. The validator rejects any SQL-looking string in structural fields. The compiler is the only SQL producer.

Lifecycle is never in a run's guarantee

lifecycle lives in the YAML but has no effect on checks — a draft contract can still be compiled (dry-run) and diffed. Only active contracts drive compliance and the cockpit Ampel.

Artifact trail

  • contracts/FACT_ORDERS.yaml — the living contract (Git)
  • contracts/FACT_ORDERS.active.yml — certified snapshot (G3 diff base)
  • checks/FACT_ORDERS/checks.yml — compiled checks (determinism header)
04 · Guarantee families

What you declare → what gets compiled

Each guarantee family is a semantic declaration. The compiler turns it into read-only SQL with a bound {schema} placeholder. Click a family to see the YAML shape and the resulting check type.

🏛️
schema
standardobs
Declares which columns must exist and whether the schema is open (new columns allowed) or closed (new columns = breaking).
guarantees: schema: columns: [ORDER_ID, AMOUNT, STATUS] mode: closed # open | closed severity: critical
→ Compiles to check type:
schema_exists type_conformance

SQL counts rows in SYS.TABLE_COLUMNS ∪ SYS.VIEW_COLUMNS. closed mode means adding a column later is a breaking change.

🔑
keys
expensivequal
Declares which column(s) must be unique. The compiler emits a duplicate-check SQL.
guarantees: keys: - columns: [ORDER_ID] unique: true severity: critical - columns: [ORDER_ID, LINE_NO] # composite key unique: true severity: critical
→ Compiles to check type:
duplicate duplicate_composite

SQL: COUNT(*) - COUNT(DISTINCT key). Gated as expensive — skipped when freshness gate fails (skipped_stale). Any change = breaking.

⏱️
freshness
gateobs
Declares the maximum age of data in the table via an ISO-8601 duration and a timestamp column.
guarantees: freshness: column: LOAD_TIMESTAMP # must be a valid identifier max_age: PT24H # ISO-8601: PT1H, P1D, PT30M … severity: warn
→ Compiles to check type:
freshness

SQL: SECONDS_BETWEEN(MAX(col), CURRENT_TIMESTAMP). Classified as gate — if it fails, expensive checks get skipped_stale. Tightening max_age = breaking.

📊
volume
gateobs
Declares a minimum row count and/or rolling baseline anomaly detection.
guarantees: volume: min_rows: 1000 # hard floor; raising this later = breaking baseline: rolling # enables rolling statistical band bounds: auto # auto-fit from history severity: warn
→ Compiles to check type:
row_count volume_anomaly

baseline: rolling is an observability signal — baselines accumulate from run history (warm-up: 5 samples). Raising min_rows = breaking.

🚫
not_null
standardqual
Declares columns that must never be NULL.
guarantees: not_null: - columns: [ORDER_ID, CUSTOMER_ID] severity: fail - columns: [NET_AMOUNT] severity: critical
→ Compiles to check type:
missing

SQL: COUNT(*) WHERE col IS NULL, expect = 0. Adding a new column to not_null on an existing contract = breaking.

📈
completeness
standardqual
Declares a minimum fill-rate (%) per column — tolerates a small null rate instead of strict zero.
guarantees: completeness: - column: EMAIL min_pct: 95.0 # at least 95 % of rows must have a value severity: warn - column: PHONE min_pct: 60.0 severity: warn
→ Compiles to check type:
completeness_pct

SQL: 100 * COUNT(NULL col) / COUNT(*), expect < (100 - min_pct). Raising min_pct or adding a column = breaking.

🔗
referential
expensivequal
Declares FK integrity — fact rows must have a matching dimension key.
guarantees: referential: - fk: [CUSTOMER_ID] parent: DIM_CUSTOMER parent_key: [CUSTOMER_ID] severity: critical
→ Compiles to check type:
reference_integrity

SQL: LEFT JOIN … WHERE dim_pk IS NULL. Classified as expensive. Removing an FK relationship = breaking for consumers.

Seed heuristics: When you call POST /api/contracts/{p}/seed, the seeder reads the inventory object, proposes schema from the column list, infers key candidates from column names ending in ID/NO/NR/KEY/CODE (marked proposed: true for steward confirmation), and adds freshness from any column containing TS/TIME/DATE/LOAD. It's a starting point — not a finished contract.

Amending an active contract (Full mode)

A PUT on an active contract always writes a draft amendment — the active version stays in .active.yml as the G3 diff baseline. The amendment only becomes active when approved. This means: an active contract can have a pending draft change without any disruption to running checks.

active → PUT (always draft) → draft → diff → approve (G3) → active ↑ snapshot updated
⬆️

Promoting an internal gate to a consumer contract

POST /api/contracts/{p}/promote copies the guarantees from an internal_gate artifact into a new consumer_contract draft (named {p}_contract). The gate and the contract co-exist; the contract carries SemVer and approval ceremony from its first certification. This is the Lite → Full transition path.

07 · Auth & permissions

Roles & write permissions

The server is authoritative — the frontend only mirrors. Write permission is determined by role × owned_by × owners membership. Filter by role to highlight what's allowed.

Action viewer steward owner admin Notes

owned_by rule

Write permission cascades: steward can write owned_by: platform contracts; owner can write both platform and product. An owners: ["grp:data-platform"] list grants fine-grained ACLs via IdP group claims (fail-closed: a missing group claim = no match).

08 · Object settings

Environment & scheduling

Two per-object settings define how and when checks run: the environment (which HANA connection to use) and the schedule (when to run automatically).

Environment

An environment is a named HANA connection. Passwords are never stored in the contract or returned to the client — only a password_ref (e.g. env:HANA_PW_PROD) is stored; the value is resolved server-side at connect time.

Connection fields

production: host: hana.example.com port: 443 user: DQ_TECHNICAL_USER schema: CENTRAL_DWH # default schema; overridable per run password_ref: env:HANA_PW_PROD encrypt: true validate_cert: true

Security rules

  • Admin-only write access to environments
  • Password value never returned to client (S-1)
  • plain: references never surfaced in the API response
  • Schema in the environment overrides the contract's dataset schema at runtime (G2)
  • ALLOW_MOCK_CONNECTION=false → no environment = 422; no silent fallback

Scheduling modes

There is at most one schedule per object (id obj:{object_id}). Requires steward+ to manage.

🖐️
manual
No automatic runs
Default state. No schedule row in the store. Runs are triggered via the UI (POST /api/objects/{id}/run) or CLI/cron by the operator. The cockpit shows the last-run timestamp so you can see how stale it is.
⚙️
internal
Signal's own poller
Signal's built-in scheduler claims the object at each interval_seconds (minimum 60s). The run path is identical to a manual trigger — same compliance/incident/notification side-effects. Good for PoC and self-contained setups.
🔌
external
External orchestrator
SAP Datasphere Task Chain, Airflow, or cron drives the cadence via cli/dq_check_runner.py. Signal's poller never claims the object. The schedule row exists for documentation and stamps last_run so the cockpit still shows it as governed.
Double-run guard (F2): The store registers a run as running before execution via a partial unique index. A second trigger while one is already running returns {"status": "already_running"} — atomically, without a check-then-act race, even across multiple workers.
